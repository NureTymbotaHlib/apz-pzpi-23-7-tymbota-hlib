Необхідно підготувати презентацію доповіді, у якій розкрити призначення патерну, описати його архітектуру, пояснити проблему прямого доступу до об’єкта, порівняти прямий доступ із використанням Proxy, а також продемонструвати приклади програмного коду, що ілюструють принцип роботи патерну.


3 ОПИС ВИКОНАНОЇ РОБОТИ
Патерн проєктування Proxy
     Патерни проєктування є типовими архітектурними рішеннями, що застосовуються для розв’язання поширених проблем під час розроблення програмного забезпечення. Вони допомагають описувати перевірені способи організації класів та об’єктів, зменшувати складність програмних систем і підвищувати зрозумілість архітектури.
     Одним із таких рішень є структурний патерн Proxy. Його основне призначення полягає у створенні об’єкта-замісника, який контролює доступ до іншого об’єкта. Клієнт взаємодіє із Proxy так само, як із реальним об’єктом, але сам Proxy може виконувати додаткові дії перед передаванням запиту реальному об’єкту.
     Патерн Proxy доцільно використовувати у випадках, коли прямий доступ до об’єкта є небажаним або потребує додаткового контролю. Наприклад, реальний об’єкт може бути ресурсомістким, створюватися тривалий час або вимагати перевірки прав доступу. У таких ситуаціях Proxy дозволяє відкласти створення реального об’єкта до моменту, коли він справді потрібен, або заборонити доступ користувачеві без необхідних прав.
Формальне визначення патерну
     Proxy визначається як структурний патерн проєктування, який надає об’єкт-замісник для контролю доступу до іншого об’єкта. Замісник реалізує той самий інтерфейс, що й реальний об’єкт, тому клієнт може працювати з ними однаково.
     Основна ідея патерну полягає у введенні проміжного об’єкта між клієнтом і реальним об’єктом. Клієнт звертається до Proxy, а Proxy самостійно визначає, чи потрібно створювати реальний об’єкт, чи має користувач право доступу, а також чи потрібно передавати запит далі.
     Таким чином, Proxy дозволяє відокремити логіку доступу від основної логіки реального об’єкта. Це робить програмну систему більш керованою, гнучкою та зручною для подальшого розширення.
Необхідність використання патерну Proxy
     Використання патерну Proxy обумовлено низкою архітектурних проблем, які можуть виникати під час прямої взаємодії клієнта з реальним об’єктом.
     По-перше, прямий доступ змушує клієнта безпосередньо працювати з конкретним класом. Це збільшує залежність між компонентами системи та ускладнює зміну внутрішньої реалізації реального об’єкта.
     По-друге, у разі прямого створення об’єкта він може завантажуватися одразу, навіть якщо фактично ще не використовується. Це є небажаним для ресурсомістких об’єктів, наприклад великих файлів, документів, зображень або об’єктів, що отримуються з віддаленого сервера.
     По-третє, прямий доступ не забезпечує централізованої перевірки прав користувача. У такому випадку логіка контролю доступу може дублюватися в різних частинах програми, що ускладнює підтримку коду.
     Патерн Proxy розв’язує ці проблеми шляхом створення проміжного об’єкта, який контролює звернення до реального об’єкта. Це дозволяє реалізувати перевірку доступу, відкладене створення об’єкта та додаткове керування його використанням.
Структура та ролі учасників патерну
     Архітектура патерну Proxy складається з кількох основних учасників, кожен із яких виконує окрему роль у взаємодії об’єктів.
     Subject є спільним інтерфейсом для реального об’єкта та його замісника. Завдяки цьому клієнт може працювати з Proxy і RealSubject однаковим способом.
     RealSubject є реальним об’єктом, який виконує основну функціональність. У розглянутому прикладі таким класом є RealDocument, який відповідає за завантаження та відображення документа.
     Proxy є об’єктом-замісником, який реалізує той самий інтерфейс, що й RealSubject. Він контролює доступ до реального об’єкта, перевіряє умови використання та створює RealSubject лише тоді, коли це необхідно.
     Client є об’єктом, який звертається до Subject. Клієнт не повинен знати, чи працює він із реальним об’єктом, чи з його замісником.
Порівняння прямого доступу та використання Proxy
     У разі прямого доступу клієнт безпосередньо створює об’єкт RealDocument і викликає його методи. Такий підхід є простим, але має певні недоліки. Зокрема, документ завантажується одразу під час створення об’єкта, а перевірка доступу користувача не виконується.
     У разі використання Proxy клієнт створює об’єкт ProxyDocument, який має той самий інтерфейс Document. Після цього всі звернення клієнта проходять через ProxyDocument. Саме цей клас перевіряє роль користувача та визначає, чи можна створювати і відображати реальний документ.
     Таким чином, прямий доступ забезпечує просту взаємодію, але не дає змоги контролювати створення та використання об’єкта. Proxy додає проміжний рівень, який дозволяє реалізувати контроль доступу та відкладене створення RealDocument.
Приклад програмної реалізації
     У межах виконання роботи було розглянуто приклад реалізації патерну Proxy мовою Java. Приклад демонструє доступ до документа з урахуванням ролі користувача.
     У програмі використовується інтерфейс Document, який містить метод display(). Цей інтерфейс реалізують два класи: RealDocument і ProxyDocument.
     Клас RealDocument є реальним об’єктом. Він зберігає назву файла, завантажує документ із диска та відповідає за його відображення. Під час створення об’єкта RealDocument викликається метод loadFromDisk(), тому документ завантажується одразу.
     Клас ProxyDocument є замісником. Він також реалізує інтерфейс Document, але перед зверненням до RealDocument виконує перевірку доступу. Якщо роль користувача не має відповідних прав, відображення документа не виконується. Якщо доступ дозволено, ProxyDocument створює RealDocument лише під час першого звернення.
Реалізація інтерфейсу Document та класу RealDocument
     Інтерфейс Document задає спільний метод display(), який повинен бути реалізований як у реальному документі, так і в його заміснику. Це забезпечує однаковий спосіб взаємодії клієнта з об’єктами.
     Клас RealDocument реалізує інтерфейс Document і містить поле fileName, у якому зберігається назва документа. Конструктор класу приймає назву файла та викликає метод loadFromDisk(). Це означає, що під час створення RealDocument документ одразу завантажується з диска.
     Метод display() відповідає за відображення документа. У прикладі він виводить повідомлення з назвою файла, що демонструє виконання основної операції реального об’єкта.
Реалізація класу ProxyDocument
     Клас ProxyDocument також реалізує інтерфейс Document, тому клієнт може використовувати його так само, як RealDocument. Основна відмінність полягає в тому, що ProxyDocument не виконує основну роботу самостійно, а контролює доступ до RealDocument.
     Клас ProxyDocument містить назву файла, роль користувача та посилання на RealDocument. На початку роботи це посилання дорівнює null, тобто реальний документ ще не створений.
     Під час виклику методу display() спочатку виконується перевірка доступу за допомогою методу hasAccess(). Якщо роль користувача не є ADMIN або TEACHER, програма виводить повідомлення про заборону доступу та завершує виконання методу.
     Якщо доступ дозволено, ProxyDocument перевіряє, чи створено об’єкт RealDocument. Якщо він ще не створений, виконується створення нового RealDocument. Після цього викликається метод display() реального документа.
Послідовність взаємодії об’єктів
     Процес взаємодії об’єктів у розглянутій реалізації відбувається у визначеній послідовності.
     Спочатку клієнт створює об’єкт ProxyDocument, передаючи назву файла та роль користувача. Далі клієнт викликає метод display() через інтерфейс Document.
     Після цього ProxyDocument виконує перевірку доступу. Якщо роль користувача дорівнює STUDENT, доступ забороняється, і реальний документ не створюється. Якщо роль користувача дорівнює TEACHER або ADMIN, доступ дозволяється.
     Після успішної перевірки ProxyDocument перевіряє, чи існує об’єкт RealDocument. Якщо він ще не створений, ProxyDocument створює його, унаслідок чого виконується завантаження документа з диска. Після цього викликається метод display() класу RealDocument.
     Під час повторного виклику display() для того самого ProxyDocument реальний документ уже існує, тому повторне завантаження не виконується. Це демонструє відкладене створення об’єкта та оптимізацію використання ресурсів.
Переваги використання патерну Proxy
     Однією з основних переваг патерну Proxy є можливість контролю доступу до реального об’єкта. У розглянутому прикладі це реалізовано через перевірку ролі користувача.
     Ще однією перевагою є відкладене створення об’єкта. RealDocument створюється лише тоді, коли доступ дозволено і документ справді потрібно відобразити. Це дозволяє уникнути зайвого використання ресурсів.
     Патерн також зменшує залежність клієнта від конкретного класу. Клієнт працює з інтерфейсом Document, тому він не залежить від того, використовується RealDocument чи ProxyDocument.
     Крім того, Proxy дозволяє додавати додаткову логіку без зміни коду реального об’єкта. Наприклад, можна реалізувати журналювання звернень, кешування, перевірку прав або роботу з віддаленими об’єктами.
Недоліки та практичні сценарії використання
     Попри переваги, патерн Proxy має певні недоліки. Основним із них є ускладнення структури програми, оскільки додається ще один клас-замісник. Це збільшує кількість коду та може ускладнити розуміння архітектури для невеликих програм.
     Також використання Proxy може додати незначні накладні витрати, оскільки кожен виклик проходить через додатковий об’єкт. У більшості випадків ці витрати є незначними, але їх потрібно враховувати у високонавантажених системах.
     Практично патерн Proxy застосовується для контролю доступу до ресурсів, відкладеного створення об’єктів, кешування даних, роботи з віддаленими сервісами, а також для журналювання звернень до об’єктів.
     У системах керування документами Proxy може використовуватися для перевірки ролей користувачів перед відкриттям файлів. У графічних редакторах він може відкладати завантаження великих зображень. У мережевих застосунках Proxy може представляти віддалений об’єкт і приховувати складність взаємодії з ним.

4 ВИСНОВКИ
     У ході виконання роботи було розглянуто патерн проєктування Proxy. Проаналізовано його призначення, основні компоненти та принципи взаємодії між ними.
     Було визначено, що Proxy є структурним патерном проєктування, який надає об’єкт-замісник для контролю доступу до реального об’єкта. Його використання дозволяє реалізувати перевірку прав користувача, відкладене створення об’єктів та додаткове керування зверненнями до ресурсів.
     У межах роботи було розглянуто приклад реалізації патерну мовою Java. У прикладі клас ProxyDocument контролює доступ до RealDocument, перевіряючи роль користувача перед відображенням документа. Також було продемонстровано, що RealDocument створюється лише після успішної перевірки доступу, що дозволяє уникнути зайвого завантаження документа.
     У результаті виконання роботи було сформовано розуміння принципів застосування патерну Proxy та його значення для побудови гнучкої, керованої та підтримуваної архітектури програмного забезпечення.

6  Створи приклад Proxy для відкладеного завантаження документа з перевіркою доступу користувача.
7  */
8
9  interface Document {
10     void display();
11 }
12
13 class RealDocument implements Document {
14     private final String fileName;
15
16     public RealDocument(String fileName) {
17         this.fileName = fileName;
18         loadFromDisk();
19     }
20
21     private void loadFromDisk() {
22         System.out.println("Завантаження документа з диска: " + fileName);
23     }
24
25     @Override
26     public void display() {
27         System.out.println("Відображення документа: " + fileName);
28     }
29 }
30
31 class ProxyDocument implements Document {
32     private final String fileName;
33     private final String userRole;
34     private RealDocument realDocument;
35
36     public ProxyDocument(String fileName, String userRole) {
37         this.fileName = fileName;
38         this.userRole = userRole;
39     }
40
41     @Override
42     public void display() {
43         if (!hasAccess()) {
44             System.out.println("Доступ заборонено для ролі: " + userRole);
45             return;
46         }
47
48         if (realDocument == null) {
49             realDocument = new RealDocument(fileName);
50         }
51
52         realDocument.display();
53     }
54
55     private boolean hasAccess() {
56         return userRole.equals("ADMIN") || userRole.equals("TEACHER");
57     }
58 }
59
60 class Main {
61     public static void main(String[] args) {
62         Document studentDocument = new ProxyDocument("architecture-report.pdf", "STUDENT");
63         studentDocument.display();
64
65         System.out.println();
66
67         Document teacherDocument = new ProxyDocument("architecture-report.pdf", "TEACHER");
68         teacherDocument.display();
69         teacherDocument.display();
70     }
71 }
