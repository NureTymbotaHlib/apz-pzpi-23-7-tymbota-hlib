Для цього було обрано горизонтальну стратегію масштабування. Її суть полягає в тому, що замість одного backend-сервера запускається декілька однакових копій застосунку, які паралельно обробляють запити користувачів.     У роботі було підготовлено два варіанти запуску: з одним backend-інстансом та з трьома backend-інстансами. Для розподілу запитів між ними використовується Nginx Load Balancer. Користувач або інструмент навантажувального тестування Locust звертається до Nginx, а він перенаправляє запити на один із доступних backend-серверів.     Усі backend-інстанси працюють з однією зовнішньою базою даних MongoDB Atlas, тому дані залишаються узгодженими незалежно від того, який саме сервер обробив запит.Загальна схема масштабування має такий вигляд:     Таким чином, система AutoInsurance може працювати як набір однакових backend-інстансів, що дозволяє розподіляти навантаження та підвищувати стабільність роботи системи.
2 ОПИС ТЕХНІЧНИХ РІШЕНЬ, ЯКІ РОБЛЯТЬ МАСШТАБУВАННЯ МОЖЛИВИМ     Основою масштабування backend-системи AutoInsurance стала контейнеризація за допомогою Docker. Для серверної частини було підготовлено Dockerfile, який дозволяє запускати backend у однаковому ізольованому середовищі.     Для запуску системи використовується Docker Compose. У роботі підготовлено дві конфігурації: docker-compose.single.yml для запуску одного backend-інстанса та docker-compose.scaled.yml для запуску трьох backend-інстансів. Це дозволяє порівняти роботу системи до масштабування та після нього.     Для балансування навантаження використовується Nginx. Він приймає HTTP-запити на одному порту та перенаправляє їх на доступні backend-інстанси. Завдяки цьому користувач працює з єдиною точкою входу, а навантаження розподіляється між кількома копіями сервера.     Усі backend-інстанси підключаються до однієї зовнішньої бази даних MongoDB Atlas. Це дозволяє зберігати спільний стан системи та уникати розбіжностей у даних між різними серверними інстансами.     Для перевірки роботи системи під навантаженням було використано Locust. За його допомогою можна імітувати одночасну роботу багатьох користувачів та порівняти кількість запитів, яку витримує система в різних конфігураціях.
3 АНАЛІЗ ВУЗЬКИХ МІСЦЬ СИСТЕМИ     Під час масштабування backend-системи AutoInsurance основними потенційними вузькими місцями є backend-інстанси, база даних, балансувальник навантаження та ресурси середовища, у якому запускаються контейнери.     Першим можливим обмеженням є CPU backend-контейнерів. Якщо кількість запитів збільшується, сервер витрачає більше ресурсів на обробку HTTP-запитів, виконання бізнес-логіки та взаємодію з базою даних. Горизонтальне масштабування частково вирішує цю проблему, оскільки запити розподіляються між кількома backend-інстансами.     Другим важливим вузьким місцем є MongoDB Atlas. Усі backend-інстанси працюють з однією базою даних, тому при великій кількості одночасних запитів саме база може стати обмеженням. На швидкість роботи можуть впливати повільні запити, кількість підключень та мережеві затримки.     Також обмеженням може бути Nginx Load Balancer, через який проходять усі запити до backend-системи. У межах цієї лабораторної роботи він не став основною проблемою, але при значному зростанні навантаження його також потрібно оптимізувати.     Ще одним фактором є локальне середовище запуску. Оскільки тестування виконується через Docker, продуктивність залежить від ресурсів комп’ютера: процесора, оперативної пам’яті та налаштувань Docker Desktop.     Отже, масштабування backend-інстансів дозволяє збільшити кількість оброблюваних запитів, але для повноцінного масштабування потрібно також враховувати продуктивність бази даних, балансувальника та середовища виконання.4 ОПИС НАВАНТАЖУВАЛЬНИХ ТЕСТІВ     Для перевірки масштабування backend-системи AutoInsurance було проведено навантажувальне тестування за допомогою інструмента Locust. Тестування виконувалося для двох конфігурацій: з одним backend-інстансом та з трьома backend-інстансами, які працюють через Nginx Load Balancer.     Під час тестування використовувалося 100 віртуальних користувачів, швидкість створення користувачів становила 10 користувачів за секунду, а тривалість тесту — 2 хвилини. Запити надсилалися до основних endpoint-ів системи: перевірка стану сервера, отримання даних про страховий поліс, отримання даних про транспортний засіб та отримання telemetry events.     Результати навантажувального тестування наведено на рисунку 4.1.Рисунок 4.1 — Результати навантажувального тестування для одного backend-інстанса     Для перевірки роботи масштабованої конфігурації було запущено три backend-інстанси AutoInsurance. Запити користувачів надходили через Nginx Load Balancer, який розподіляв навантаження між трьома однаковими копіями backend-сервера. Параметри тестування залишилися такими самими, як і для одного інстанса: 100 віртуальних користувачів, швидкість запуску — 10 користувачів за секунду, тривалість тесту — 2 хвилини.     Результати навантажувального тестування для трьох backend-інстансів наведено на рисунку 4.2.Рисунок 4.2 — Результати навантажувального тестування для трьох backend-інстансів     На рисунку 4.2 наведено результати тестування конфігурації з трьома backend-інстансами. Під час тесту було виконано 12304 запити без помилок. Середня пропускна здатність системи становила 102.64 запита за секунду, середній час відповіді — 38.63 мс, а показник 95%ile — 120 мс.     Порівняно з конфігурацією з одним backend-інстансом, масштабована конфігурація показала трохи кращий середній час відповіді: 38.63 мс проти 40.56 мс.
